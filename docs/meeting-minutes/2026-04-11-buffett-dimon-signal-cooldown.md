# Buffett x Dimon 토론: 시그널 중복 방지 5분 쿨다운 제거 여부

**일시**: 2026-04-11  
**참석**: Warren Buffett (자본 보존 최우선), Jamie Dimon (기관 리스크 관리)  
**주제**: 같은 시그널(type + direction) 5분 쿨다운을 제거할 것인가?  
**배경**: 스캐너 1초 주기, `max_open_positions=1`, 다층 게이트(counter-trend, velocity brake, daily loss 등) 존재

---

## 문제 정의

현행 로직:
```python
sig_key = f"{signal['type']}_{signal['direction']}"
if sig_key in self._recent_signals:
    if now - self._recent_signals[sig_key] < 300_000:  # 5분
        return None
```

| 시나리오 | 현행 동작 | 문제 |
|---------|----------|------|
| 시그널 → counter-trend 게이트 차단 | 5분간 재시도 불가 | 30초 후 상위 TF 방향 변경되면 기회 상실 |
| 시그널 → 포지션 진입 성공 | 5분간 같은 시그널 무시 | 문제 없음 (이미 `max_open_positions=1`로 보호) |
| 시그널 → velocity brake 차단 | 5분간 재시도 불가 | brake 해제 후에도 대기 필요 |
| 쿨다운 제거 시 | 1초마다 같은 시그널 재평가 | API/로그 부하 우려 |

---

## Round 1: 기본 입장

### Warren Buffett (조건부 제거 찬성)

> "5분 쿨다운이 처음 도입된 이유는 이해하네. 같은 시그널이 1초마다 주문을 쏘는 걸 막으려 했지. 하지만 지금 시스템을 보면, 주문을 실제로 넣기까지 최소 6개의 게이트를 통과해야 하네."

- **인정하는 가치**: 쿨다운 없이 1초마다 같은 시그널을 처리하면 불필요한 연산과 로그가 쌓임
- **더 큰 문제**: counter-trend 게이트에서 막힌 유효 시그널이 5분 동안 재시도 불가. 시장은 5분 사이에 상위 TF 캔들이 바뀌고, 트렌드 분류가 COUNTER_TREND → WITH_TREND로 전환될 수 있음
- **조건**: 완전 제거보다는 **쿨다운을 대폭 줄이되, 최소한의 스로틀은 유지**해야 함. 1초마다 전체 게이트를 평가하는 건 불필요한 리소스 낭비

### Jamie Dimon (완전 제거 찬성)

> "기관 시스템에서 시그널 쿨다운이라는 개념 자체가 없어. 시그널은 시그널이고, 리스크 게이트가 허용하면 실행하는 거야. 시그널을 인위적으로 억제하는 건 리스크 관리가 아니라 기회 차단이지."

- **기관 관행**: 시그널 생성과 주문 실행은 별개 레이어. 시그널을 필터링하는 건 리스크 엔진의 역할이지 시그널 계층의 역할이 아님
- **현재 방어선 점검**:
  - `max_open_positions=1` → 이미 열린 포지션이면 진입 불가
  - 같은 방향이면 SL 조임만 수행, 새 포지션 안 열림
  - velocity brake → 3연속 SL 시 30분 중단
  - daily loss tier → -3%에서 사이즈 절반, -5%에서 당일 중단
  - counter-trend 게이트 → 상위 TF 기반 필터링
- **결론**: 이 모든 게이트가 이미 존재하므로 5분 쿨다운은 **중복 방어**이며, 오히려 해로움

## Round 2: 반론과 우려

### Warren Buffett (로그 폭탄 우려)

> "Jamie, 자네 말에 일리가 있네. 하지만 실무적 문제를 생각해보게. 스캐너가 1초마다 돌면서 confluence 시그널이 연속으로 뜨면, 같은 시그널이 300번(5분간) 게이트를 두드리는 거야. 포지션이 이미 열려있어서 매번 `return None`하겠지만, 로그가 초당 1줄씩 쌓이네."

- **계산**: 1초마다 × 60초 × 60분 × 24시간 = 하루 86,400줄의 동일 시그널 로그
- **제안**: 완전 제거 대신 **10초 쿨다운**. 이러면:
  - 시장 조건 변화(30초~1분)에 충분히 빠르게 반응
  - 로그 양을 1/10로 줄임 (8,640줄/일)
  - 1초 연타 주문 시도 원천 차단

### Jamie Dimon (10초는 너무 길다, 5초 제안)

> "10초도 괜찮지만, 우리 스캐너가 1초 주기잖아. 30m 캔들이 닫히는 순간 트렌드 분류가 바뀔 수 있고, 그 직후 시그널이 들어오면 5초면 충분히 빠르게 재평가할 수 있어."

- **실질적 시나리오**: 
  1. 14:00:00 — `confluence_bullish` 시그널 → counter-trend 차단
  2. 14:00:01 — 30m 캔들 종가 확정 → 상위 TF 방향 전환
  3. 14:00:05 — 같은 시그널 재평가 → WITH_TREND → 진입 성공
- **5초면 충분한 이유**: 캔들 데이터 갱신 주기가 1초이므로, 5초면 최소 5번의 갱신 사이클을 거침
- **API 부하**: Binance 주문 API는 시그널이 모든 게이트를 통과할 때만 호출. 게이트 평가는 순수 인메모리 연산이므로 5초 주기면 부하 무시 가능

## Round 3: 합의 도출

### Warren Buffett (5초 수용, 조건 추가)

> "5초를 받아들이겠네. 다만 한 가지 조건: **시그널이 게이트를 통과해서 실제 주문이 실행된 경우에는 별도로 30초 쿨다운**을 걸어야 하네. 주문 실행 직후 같은 시그널이 5초 만에 다시 주문을 넣으려 하면 안 되니까."

- **분석**: 이 우려는 사실 불필요. `max_open_positions=1`이므로 주문 성공 후 같은 방향 시그널은 SL 조임만 수행하고 새 주문 안 냄. 반대 방향은 교체 로직(20분 쿨다운, PnL 조건)이 별도로 존재
- **자기 반론 수용**: 그렇다면 5초 단일 쿨다운으로 충분. 추가 조건 철회

### Jamie Dimon (최종 동의)

> "좋아. 5초 쿨다운은 **스로틀**의 역할만 하고, 실질적 리스크 관리는 기존 게이트들이 담당하는 구조야. 이게 올바른 관심사 분리(separation of concerns)지."

## 최종 합의

### 변경 사항

| 항목 | 현행 | 변경 |
|-----|------|------|
| 시그널 중복 쿨다운 | **300,000ms (5분)** | **5,000ms (5초)** |
| 만료 정리 주기 | 600,000ms (10분) | **60,000ms (1분)** |

### 변경 코드

```python
# 시그널 스로틀 (5초)
expired = [k for k, t in self._recent_signals.items() if now - t > 60_000]
for k in expired:
    del self._recent_signals[k]
sig_key = f"{signal['type']}_{signal['direction']}"
if sig_key in self._recent_signals:
    if now - self._recent_signals[sig_key] < 5_000:  # 5초
        return None
self._recent_signals[sig_key] = now
```

### 근거

1. **기회 보존**: 5분 → 5초로 줄여 게이트 조건 변화에 빠르게 반응 (최대 5초 지연)
2. **중복 방어 불필요**: `max_open_positions=1`, 같은 방향 SL 조임, 교체 20분 쿨다운 등 6개 게이트가 이미 존재
3. **로그/연산 절제**: 1초마다 평가하지 않고 5초 스로틀로 불필요한 반복 제거
4. **리스크 영향 없음**: 실제 주문 실행까지의 모든 리스크 게이트는 그대로 유지

### 투표

- Warren Buffett: **찬성** (5초 스로틀 + 기존 게이트 유지)
- Jamie Dimon: **찬성** (시그널 억제 최소화, 리스크 게이트에 위임)
