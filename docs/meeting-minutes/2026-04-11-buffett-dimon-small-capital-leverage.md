# Buffett x Dimon 토론: 소자본 분할 진입 vs Binance 최소 노셔널 충돌

**일시**: 2026-04-11  
**참석**: Warren Buffett (자본 보존 최우선), Jamie Dimon (기관 리스크 관리)  
**주제**: $200 소자본에서 분할 진입 tranche가 Binance 최소 노셔널($100)에 걸려 주문 실패하는 문제 해결  
**배경**: 현행 WITH_TREND 50/30/20 분할에서 30%, 20% tranche가 노셔널 미달로 자동 취소됨

---

## 문제 정의

### 현행 수치 (BTC $84,000 기준)

```
자본: $200 USDT
레버리지: 5x 고정
리스크: $200 x 2% = $4
SL distance: ~0.87% (1.2 x ATR 기준)
포지션 노셔널: $4 / 0.87% = ~$460
total_qty: $460 / $84,000 = 0.005 BTC
```

### 분할 시 각 tranche

| Tranche | Split | 수량 | 노셔널 | Binance 최소 | 결과 |
|---------|-------|------|--------|-------------|------|
| 0 (50%) | 0.50 | 0.003 BTC | $252 | $100 | 체결 |
| 1 (30%) | 0.30 | 0.001 BTC | $84 | $100 | **실패** |
| 2 (20%) | 0.20 | 0.001 BTC | $84 | $100 | **실패** |

**핵심**: BTC 최소 수량 단위가 0.001이라 0.0015 BTC(30%)와 0.001 BTC(20%)가 모두 0.001로 내림되어 노셔널 $84. Tranche 1, 2가 `CANCELLED` 처리되어 사실상 **100% 즉시 진입**과 동일. 분할 진입의 평단 최적화/보호 효과가 완전히 상실됨.

---

## 검토 방안

| 방안 | 설명 | 장점 | 단점 |
|------|------|------|------|
| A. 레버리지 10x | 같은 2% 리스크로 노셔널 2배 | tranche 노셔널 증가 | 청산가 접근, 20% tranche 여전히 $97 미달 가능 |
| B. 50/50 2분할 | 비율 재조정 | 둘 다 $100+ 확실 | 3단계 평단 최적화 상실 |
| C. 100% 즉시 진입 | 분할 포기 | 단순, 노셔널 문제 없음 | 역행 시 보호 없음 |
| D. 자본 기반 자동 분할 수 결정 | 각 tranche >= $100 보장 | 모든 규모에서 최적 | 구현 복잡 |

## Round 1: 기본 입장

### Warren Buffett (방안 D 지지 -- 자동 분할 수 조정)

> "레버리지를 올려서 문제를 해결한다? 그건 빚을 더 내서 빚을 갚겠다는 것과 같네. 우리가 4월 10일에 레버리지 5x로 합의한 이유를 잊었나? 청산가가 가까워지면 변동성 하나에 전체 잔고가 날아갈 수 있어."

- **레버리지 5x vs 10x 청산가 비교** (BTC $84,000, Cross Margin):

| 레버리지 | 마진 | 100% 청산 거리 | 청산가 (LONG) | 청산가 (SHORT) |
|---------|------|--------------|-------------|-------------|
| 5x | $92 | ~20% | ~$67,200 | ~$100,800 |
| 10x | $46 | ~10% | ~$75,600 | ~$92,400 |

- **Cross margin이라 전체 잔고($200)가 마진** → 실제 청산 거리는 더 넓음
- **하지만**: 10x에서 급변동 시 unrealized loss가 잔고를 빠르게 잠식. 2~3포지션 연속 역행하면 마진콜 위험 증가
- **방안 A의 치명적 결함**: 20% tranche 노셔널 = $460 x 2 x 20% = $184... 아니, 수량이 0.001 BTC 단위로 내림됨. 0.002 BTC = $168이면 충족이지만, ATR이 크면 포지션이 작아져 여전히 미달 가능. **근본적 해결이 아님**

> "내 원칙은 명확해. 문제의 근본은 노셔널이 $100 미만인 거야. 해결책은 각 tranche가 $100 이상이 되도록 분할 수를 줄이는 거지, 레버리지를 올리는 게 아니야."

### Jamie Dimon (방안 D 지지, 방안 B 병행)

> "기관에서 소규모 펀드가 파생상품을 거래할 때도 같은 문제가 생겨. 최소 계약 단위 때문에 분할 실행이 안 되면, **분할 수를 줄이는 게 표준**이야. 레버리지를 올리는 건 절대 아니지."

- **기관 사례**: $10M 펀드가 S&P 500 선물(1계약 = ~$250K) 분할 매수 시, 4분할이 안 되면 2분할 또는 일괄 진입으로 전환. 레버리지는 리스크 파라미터로 관리하지, 주문 단위 문제 해결용이 아님
- **방안 A 정량 분석**:

```
레버리지 10x, 리스크 $4, SL 0.87%
포지션 노셔널: $4 / 0.87% = $460 (변동 없음!)
```

> "잠깐. 여기서 중요한 걸 짚겠네. **SL 기반 사이징에서는 레버리지를 올려도 노셔널이 변하지 않아.** 노셔널은 risk / SL_distance로 결정되고, 레버리지는 여기에 관여하지 않아. 레버리지가 바뀌는 건 **필요 마진**뿐이지."

**Buffett**: "Jamie가 핵심을 찔렀네! 레버리지 10x로 올려봤자 노셔널은 $460 그대로야. tranche 노셔널도 동일하게 $84. **방안 A는 아예 문제를 해결하지 못해.**"

### 방안 A 완전 기각 -- 수치적 증명

```
현행 (5x):
  노셔널 = risk / sl_distance = $4 / 0.87% = $460
  마진 = $460 / 5 = $92
  30% tranche 노셔널 = $460 x 30% → 0.001 BTC = $84 (미달)

방안 A (10x):
  노셔널 = risk / sl_distance = $4 / 0.87% = $460 (동일!)
  마진 = $460 / 10 = $46
  30% tranche 노셔널 = $460 x 30% → 0.001 BTC = $84 (여전히 미달!)
```

**SL 기반 사이징에서 레버리지는 노셔널에 영향을 주지 않는다. 마진만 줄어든다.**

## Round 2: 방안 D 구체 설계

### Jamie Dimon (알고리즘 제안)

> "자동 분할 수 결정 로직을 구체화하겠네. 핵심은 각 tranche의 노셔널이 Binance 최소($100)를 넘기되, 최소 수량 단위(0.001 BTC)에 의한 내림도 반영해야 해."

**알고리즘**:

```python
def determine_tranches(total_notional: Decimal, current_price: Decimal, 
                       min_notional: Decimal = Decimal("100"),
                       min_qty_step: Decimal = Decimal("0.001")) -> tuple[list[float], list[float]]:
    """
    총 노셔널과 현재가를 기준으로 분할 수와 비율을 자동 결정.
    Returns: (offsets, splits)
    """
    # 최소 수량의 노셔널 = min_qty_step x current_price
    min_qty_notional = min_qty_step * current_price  # e.g., 0.001 x 84000 = $84
    
    # 최소 tranche 노셔널 = max(Binance $100, 최소수량 노셔널)
    effective_min = max(min_notional, min_qty_notional)
    
    # 3분할 (50/30/20) 가능?
    if total_notional * Decimal("0.20") >= effective_min:
        return [0.0, -0.3, -0.6], [0.50, 0.30, 0.20]
    
    # 2분할 (60/40) 가능?
    if total_notional * Decimal("0.40") >= effective_min:
        return [0.0, -0.3], [0.60, 0.40]
    
    # 2분할 (50/50) 가능?
    if total_notional * Decimal("0.50") >= effective_min:
        return [0.0, -0.3], [0.50, 0.50]
    
    # 분할 불가 → 100% 즉시 진입
    return [0.0], [1.0]
```

**규모별 시뮬레이션** (BTC $84,000, SL 0.87%, 5x):

| 자본 | 리스크(2%) | 노셔널 | 20% tranche | 40% tranche | 50% tranche | 결정 |
|------|-----------|--------|------------|------------|------------|------|
| $200 | $4 | $460 | $84 | $184 | $230 | **2분할 (60/40)** |
| $300 | $6 | $690 | $138 | $276 | $345 | **3분할 (50/30/20)** |
| $500 | $10 | $1,150 | $230 | $460 | $575 | **3분할 (50/30/20)** |
| $150 | $3 | $345 | $69 | $138 | $172 | **2분할 (60/40)** |
| $100 | $2 | $230 | $46 | $92 | $115 | **2분할 (50/50)** |
| $70 | $1.4 | $161 | $32 | $64 | $80 | **1분할 (100%)** |

### Warren Buffett (검증 및 보완)

> "Jamie의 알고리즘은 좋네. 하지만 두 가지를 보강해야 해."

**보강 1: COUNTER_TREND도 적용**

- 현행 COUNTER_TREND: 60/40 2분할. $200에서 40% tranche = $184 → 충족
- 하지만 자본 $100이면: 40% = $92 → 미달
- COUNTER_TREND도 같은 로직 적용 필요

**보강 2: 로그 + 알림**

> "분할 수가 자동으로 줄어들 때 반드시 로그에 남기고, Telegram으로도 알려야 해. 트레이더가 '왜 분할이 안 됐지?' 하고 혼란스러워하면 안 되네."

```
[LIVE] Auto-adjusted tranches: 3 → 2 (60/40) due to min notional $100
       Notional: $460, min tranche: $184 (60/40), $92 would be below $100 (50/30/20)
```

### Jamie Dimon (동의 + 추가)

> "Buffett 말이 맞아. 한 가지 더 -- BTC 가격 변동에 따라 min_qty_notional도 바뀌어. BTC $100,000이면 0.001 BTC = $100으로 딱 경계선이야. $120,000이면 0.001 = $120이라 effective_min이 올라가지. 알고리즘이 이미 이걸 반영하고 있으니 좋아."

### 공동 성명

> **Buffett**: "레버리지는 도구야, 해결책이 아니야. 오늘의 핵심 발견은 **SL 기반 사이징에서 레버리지를 올려도 노셔널이 변하지 않는다**는 사실이야. 이건 직관에 반하지만 수학적으로 명확해. 해결책은 거래소 제약에 맞춰 분할 수를 유연하게 조정하는 거지."
>
> **Dimon**: "동의해. 그리고 이건 일시적 문제야. $200에서 시작해 $300만 넘기면 3분할이 가능해져. 자본이 성장하면 자연스럽게 더 정교한 분할이 활성화되는 시스템이야. 이게 올바른 설계 방향이지."

### 합의 결과

#### 1. 확정 파라미터

| 파라미터 | 이전 값 | **새 값** | 변경 사유 |
|---------|---------|----------|----------|
| 레버리지 | 5x 고정 | **5x 고정 (유지)** | 레버리지 변경은 노셔널에 무영향, 기각 |
| WITH_TREND 분할 | 고정 50/30/20 | **자동 결정** | 자본/노셔널 기반 |
| COUNTER_TREND 분할 | 고정 60/40 | **자동 결정** | 동일 로직 적용 |
| 분할 결정 기준 | 없음 | **각 tranche 노셔널 >= $100** | Binance 최소 노셔널 |

#### 2. 자동 분할 매핑 (WITH_TREND)

| 조건 | 분할 | Offsets | Splits |
|------|------|---------|--------|
| 최소 tranche(20%) >= $100 | 3분할 | [0.0, -0.3, -0.6] | [0.50, 0.30, 0.20] |
| 최소 tranche(40%) >= $100, 20% < $100 | 2분할 | [0.0, -0.3] | [0.60, 0.40] |
| 최소 tranche(50%) >= $100, 40% < $100 | 2분할 | [0.0, -0.3] | [0.50, 0.50] |
| 50% < $100 | 1분할 | [0.0] | [1.0] |

#### 3. 자동 분할 매핑 (COUNTER_TREND)

| 조건 | 분할 | Offsets | Splits |
|------|------|---------|--------|
| 최소 tranche(40%) >= $100 | 2분할 | [0.0, -0.3] | [0.60, 0.40] |
| 40% < $100, 50% >= $100 | 2분할 | [0.0, -0.3] | [0.50, 0.50] |
| 50% < $100 | 1분할 | [0.0] | [1.0] |

#### 4. 구현 위치

```python
# engine.py 또는 live_engine.py — on_signal() 내
# _create_entry_tranches() 호출 전에 분할 수 결정

def _determine_entry_plan(
    self, total_notional: Decimal, current_price: Decimal,
    is_counter: bool = False,
) -> tuple[list[float], list[float]]:
    """자본 규모에 따라 분할 수/비율을 자동 결정."""
    min_notional = self.settings.min_notional  # $100
    min_qty_step = Decimal("0.001")
    min_qty_notional = min_qty_step * current_price
    effective_min = max(min_notional, min_qty_notional)
    
    if is_counter:
        # COUNTER_TREND: 최대 2분할
        if total_notional * Decimal("0.40") >= effective_min:
            return [0.0, -0.3], [0.60, 0.40]
        if total_notional * Decimal("0.50") >= effective_min:
            return [0.0, -0.3], [0.50, 0.50]
        return [0.0], [1.0]
    else:
        # WITH_TREND: 최대 3분할
        if total_notional * Decimal("0.20") >= effective_min:
            return [0.0, -0.3, -0.6], [0.50, 0.30, 0.20]
        if total_notional * Decimal("0.40") >= effective_min:
            return [0.0, -0.3], [0.60, 0.40]
        if total_notional * Decimal("0.50") >= effective_min:
            return [0.0, -0.3], [0.50, 0.50]
        return [0.0], [1.0]
```

#### 5. 로깅 요구사항

- 분할 수가 기본값(3 또는 2)에서 축소될 때 INFO 로그
- Telegram 알림에 실제 적용된 분할 수 표시
- 예: `"Entry: 2 tranches (60/40) -- auto-adjusted from 3 due to min notional"`

### Buffett 부대조건

1. **레버리지 5x 절대 고정** -- 이번 논의에서 "레버리지로 해결"이 불가능함이 수학적으로 증명됨. 향후 레버리지 변경 제안 시 이 회의록을 참조할 것
2. **자본 $300 도달 시 자동으로 3분할 활성화** -- 별도 수동 조작 불필요, 알고리즘이 자동 처리

### Dimon 부대조건

1. **분할 축소 로그는 반드시 남길 것** -- 추후 성과 분석 시 "1분할이었던 거래"와 "3분할이었던 거래"의 승률/평균 수익 비교 가능
2. **BTC 가격 상승에 따른 min_qty_notional 변화 모니터링** -- BTC $100,000+에서 0.001 BTC = $100이 경계선이 됨

## 핵심 인사이트

> **Buffett**: "오늘 가장 중요한 교훈은 이거야 -- 'SL 기반 사이징에서 레버리지를 올려도 포지션 크기(노셔널)는 변하지 않는다.' 많은 트레이더가 이걸 오해하고 레버리지를 올려서 더 큰 포지션을 잡으려 하지. 하지만 리스크를 고정한 순간, 레버리지는 필요 마진만 바꿀 뿐이야."

> **Dimon**: "소자본의 제약은 레버리지로 극복하는 게 아니라, 전략을 제약에 맞추는 거야. $200로 시작하면 2분할로 충분해. $300이 되면 자동으로 3분할이 열려. 시스템이 자본 성장에 따라 스스로 진화하는 거지."

## 결론

**레버리지 변경 기각**. SL 기반 사이징에서 레버리지는 노셔널에 영향 없음 (수학적 증명).  
**자동 분할 수 조정 채택**. 각 tranche 노셔널 >= $100 보장하도록 분할 수를 자본 규모에 따라 자동 결정.  
**현재 $200 자본**: WITH_TREND 2분할(60/40), COUNTER_TREND 2분할(60/40).  
**$300+ 도달 시**: 자동으로 WITH_TREND 3분할(50/30/20) 활성화.
